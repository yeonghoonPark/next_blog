## HTTP 상태 코드란

[# 목차](#-목차)

HTTP 상태 코드는 클라이언트와 서버 간의 요청-응답 과정에서 서버가 클라이언트에게 요청 처리 상태를 알려주는 세 자리 숫자로 이루어진 코드입니다.
이 코드는 HTTP 응답의 상태 라인<sup>(Status Line)</sup>에 포함되며 HTTP 프로토콜의 중요한 부분으로,
클라이언트가 서버에 요청하고 응답을 받을 때 상태 코드가 함께 전송됩니다.
_상태 코드는 요청의 결과가 성공적인지 실패한 것인지를 판단하는 중요한 지표가 됩니다._

상태 코드는 5개의 범주로 나뉘며, 각 범주는 요청 처리의 상태를 나타냅니다.

1. _100번 대: 정보 제공 (Response)_
2. _200번 대: 성공 (Success)_
3. _300번 대: 리다이렉션 (Redirection)_
4. _400번 대: 클라이언트 오류 (Client Error)_
5. _500번 대: 서버 오류 (Server Error)_

각각의 코드 범주는 서로 다른 상황을 나타내며, 클라이언트와 서버 간의 원활한 통신을 위한 중요한 역할을 합니다.

## 100번 대: 정보 제공 (Response)

[# 목차](#-목차)

_100번 대 상태 코드는 클라이언트에게 서버가 요청을 처리하고 있다는 사실을 알리기 위한 용도로 사용됩니다._
이 상태 코드는 주로 완전히 처리되기 전의 중간 상태를 나타내며, 클라이언트에게 계속해서 요청을 보내도 된다는 신호를 보내는 역할을 합니다.
즉, 서버가 요청을 받았으며, 요청 처리 과정에서 문제가 없음을 알려주는 것입니다.

### 100번 대 주요 상태 코드

- _100 Continue_  
  클라이언트가 요청을 보내면, 서버는 일부를 먼저 처리하고, 요청을 계속 진행할 수 있다는 신호를 보냅니다.
  예를 들어, 클라이언트가 큰 파일을 업로드하려 할 때, 서버는 일단 파일의 일부를 받았고 나머지 파일을 계속 전송할 수 있다는 응답 상태 코드로 사용합니다.

- _101 Switching Protocols_  
  _클라이언트가 서버에게 특정 프로토콜을 변경할 수 있도록 요청할 때 사용됩니다._
  서버가 이 요청을 수락하여 프로토콜을 변경하는 상황에서 사용됩니다.
  예를 들어, 클라이언트가 웹 브라우저를 열고 서버와 HTTP로 통신하다가, 실시간 데이터를 주고받기 위해 WebSocket을 사용하고 싶다는 요청을 할 때,
  클라이언트는 `Upgrade` 헤더를 포함한 요청을 보내고, 서버는 이를 수락하여 HTTP 프로토콜을 WebSocket 프로토콜로 전환합니다.
  이때, 서버는 응답 상태 코드와 메세지로 `101 Switching Protocols` 를 보냅니다.

- _102 Processing_  
  _서버가 클라이언트의 요청을 처리 중인 상태에서 시간이 오래 걸리는 경우 사용됩니다._
  이 상태 코드는 주로 WebDAV 확장에서 사용되며 요청을 받았지만, 처리에 시간이 오래 걸리는 경우, 클라이언트에게 처리가 진행 중임을 알리기 위해 사용됩니다.
  예를 들어, 서버가 긴 계산을 하거나 대용량 데이터를 처리하는 중일 때 클라이언트에게 "처리가 진행 중"이라고 알리기 위해 102 Processing 응답을 보냅니다.

> _WebDAV (Web Distributed Authoring and Versioning)_  
> WebDAV는 HTTP 프로토콜을 확장하여 _파일의 원격 관리를 가능하게 해주는 기술입니다._
> 이를 통해 파일 서버와 파일 시스템을 연결하여, 로컬 컴퓨터에서 마치 로컬 파일처럼 서버에 있는 파일을 업로드, 다운로드, 수정, 삭제 등으로 조작할 수 있습니다.
> 클라우드 스토리지, 파일 서버 관리 등의 분야에서 활용되며 Google Drive, OneDrive와 같은 클라우드 스토리지 서비스가 WebDAV 프로토콜을 활용하여 파일 관리 기능을 제공합니다.

## 200번 대: 성공 (Success)

[# 목차](#-목차)

_200번 대 상태 코드는 클라이언트의 요청이 정상적으로 처리되어 성공적으로 응답이 반환되었음을 나타냅니다._
이 범주에 속하는 상태 코드들은 서버가 클라이언트의 요청을 제대로 처리했으며, 요청이 성공적으로 수행되었음을 알리는 신호로 사용됩니다.
200번 대 상태 코드는 클라이언트에게 요청에 대한 성공적인 결과를 전달하는 역할을 합니다.

### 200번 대 주요 상태 코드

- _200 OK_  
  _클라이언트의 요청이 성공적으로 처리되었음을 나타내는 가장 기본적인 상태 코드입니다._
  서버가 클라이언트의 요청을 정상적으로 처리하고, 그 결과를 응답 본문에 담아 반환할 때 사용됩니다.
  `GET`, `POST`, `PUT` 등의 요청에서 흔히 사용됩니다.
  예를 들어, 클라이언트가 웹 페이지를 요청했을 때, 서버가 해당 페이지를 정상적으로 반환하면 `200 OK` 상태 코드가 응답됩니다.

- _201 Created_  
  _클라이언트의 요청이 성공적으로 처리되었으며, 그 결과로 새로운 리소스가 생성되었음을 나타내는 상태 코드입니다._
  보통 `POST` 요청을 통해 리소스를 생성할 때 사용됩니다.
  생성된 리소스의 위치는 응답 헤더의 `Location` 필드에 포함될 수 있습니다.
  예를 들어, 클라이언트가 새로운 사용자를 서버에 등록하는 요청을 보냈을 때, 서버가 새로운 사용자를 성공적으로 생성한 경우 `201 Created` 상태 코드가 반환됩니다.

- _202 Accepted_  
  _클라이언트의 요청이 수락되었으나 처리되지 않은 상태임을 나타냅니다._
  즉, 서버는 요청을 받은 후 비동기적으로 처리할 것임을 의미합니다. 요청 자체는 정상적으로 수락되었으나, 실제 처리는 나중에 이루어질 수 있습니다.
  이 상태 코드는 주로 긴 처리 시간이 필요한 작업에서 사용됩니다.
  예를 들어, 클라이언트가 서버에 대용량 파일을 업로드하는 요청을 보냈을 때,
  서버는 해당 요청을 수락하고 백그라운드에서 처리하기 시작하지만, 즉시 처리 결과를 반환하지 않고 대신 `202 Accepted` 상태 코드를 반환합니다.

- _204 No Content_  
  _서버가 요청을 성공적으로 처리했으나, 응답 본문<sup>(Body)</sup>에 내용이 없다는 의미입니다._
  이는 주로 `DELETE` 요청 후 리소스를 성공적으로 삭제했을 때, 또는 `PUT` 요청 후 데이터를 수정했을 때 사용됩니다.
  클라이언트에게 응답 본문이 필요 없는 경우에 사용됩니다.
  예를 들어, 클라이언트가 `PUT`, `DELETE` 요청을 보내서 리소스를 수정하거나 삭제했을 때,
  서버는 리소스 수정 또는 삭제가 성공적으로 처리되었음을 나타내기 위해 `204 No Content`를 응답으로 보낼 수 있습니다.

## 300번 대: 리다이렉션 (Redirection)

[# 목차](#-목차)

_300번 대 상태 코드는 클라이언트가 요청한 리소스의 위치가 변경되었거나, 클라이언트가 요청을 다른 URL로 리다이렉션해야 함을 나타냅니다._
리다이렉션이 발생하면, 서버는 클라이언트에게 요청한 리소스의 새로운 위치를 알려주고, 클라이언트는 서버가 제공한 새로운 URL로 요청을 재전송해야 합니다.
즉, 클라이언트가 해당 리소스를 다시 요청할 때 새로운 경로로 접근하도록 유도하는 방식입니다.

### 300번 대 주요 상태 코드

- _301 Moved Permanently_  
  _클라이언트가 요청한 리소스가 영구적으로 다른 위치로 이동되었음을 나타냅니다._
  이 상태 코드는 리소스의 URL이 영구적으로 변경되었을 때 사용됩니다.
  클라이언트는 이후부터 새로운 URL로 요청을 보내야하며, 서버는 응답 헤더의 `Location` 필드에 새로운 URL을 포함시킵니다.
  예를 들어, 웹 사이트의 주소가 www.old-site.com 에서 www.new-site.com 로 영구적으로 변경된 경우,
  서버는 `301 Moved Permanently` 상태 코드와 새로운 URL을 포함시키며, 클라이언트는 새로운 URL을 기억하고 이후 모든 요청에 대해 새로운 URL을 사용해야 합니다.

- _302 Found_  
  _클라이언트가 요청한 리소스가 임시적으로 다른 위치에 있음을 나타냅니다._
  이 상태 코드는 리소스의 이동이 일시적일 때 사용됩니다. 클라이언트는 새로운 URL로 요청을 보내야 하지만,
  이 리다이렉션은 임시적이므로 이후에도 원래의 URL로 요청할 수 있습니다.
  예를 들어, 웹 사이트의 유지보수나 트래픽 분산을 위해 임시적으로 다른 URL로 리다이렉션하는 경우 사용됩니다.

- _304 Not Modified_  
  _클라이언트가 요청한 리소스가 변경되지 않았음을 나타냅니다._
  클라이언트가 헤더에 조건부 요청을 포함하여 보냈을 때, 서버는 해당 리소스가 수정되지 않은 경우 `304 Not Modified`를 반환합니다.
  클라이언트는 서버로부터 새로운 데이터를 받지 않고 캐시 된 데이터를 그대로 사용할 수 있습니다.
  예를 들어, 클라이언트가 이미 캐시 된 이미지나 페이지를 다시 요청했을 때, 서버가 해당 리소스가 변경되지 않았음을 확인하고 `304 Not Modified`를 반환하여 클라이언트가 캐시 된 데이터를 사용할 수 있도록 합니다.

- _307 Temporary Redirect_
  _리소스가 임시적으로 다른 위치로 이동되었음을 나타냅니다._
  클라이언트는 요청 메서드를 변경하지 않고 동일한 메서드로 리다이렉션을 수행해야 합니다.
  _이 상태 코드는 `302 Found`와 유사하지만, HTTP 메서드가 변경되지 않도록 보장하는 점이 다릅니다._
  예를 들어, 클라이언트가 `POST` 요청을 보냈을 때, 서버는 `307 Temporary Redirect` 상태 코드와 새로운 URL을 반환하며,
  클라이언트는 같은 `POST` 메서드로 리다이렉션을 따라가야 합니다.

## 400번 대: 클라이언트 오류 (Client Error)

[# 목차](#-목차)

_400번 대 상태 코드는 클라이언트 측에서 발생한 오류를 나타내는 코드들입니다._
즉, 클라이언트가 서버에 보낸 요청에 문제가 있을 때 발생하며, 서버는 요청을 처리할 수 없거나 이해할 수 없다는 응답을 보냅니다.
클라이언트가 오류를 수정하고 요청을 다시 보내야 하는 경우이며, 이러한 오류는 보통 잘못된 문법, 인증 문제, 요청 파라미터 누락 등
클라이언트가 수정할 수 있는 문제로 인해 발생합니다.

### 400번 대 주요 상태 코드

- _400 Bad Request_  
  _서버가 클라이언트의 요청을 이해할 수 없거나 잘못된 요청을 받았을 때 사용됩니다._
  요청이 잘못되었거나 형식이 올바르지 않으면 서버는 `400 Bad Request`를 반환합니다.
  이는 잘못된 문법이나 필수 매개변수 누락 등 다양한 이유로 발생할 수 있습니다.
  예를 들어, 클라이언트가 잘못된 JSON 형식으로 데이터를 보내거나, 요청의 필수 매개변수가 누락되었을 때 사용됩니다.

- _401 Unauthorized_  
  _클라이언트가 요청한 리소스에 접근하기 위한 인증을 제공하지 않았을 때 사용됩니다._
  이 상태 코드는 인증이 필요한 리소스에 접근하려 할 때, 클라이언트가 적절한 인증 정보를 제공하지 않은 경우(사용자 인증 실패, 인증 토큰 누락)에 발생합니다.
  예를 들어, 로그인하지 않은 사용자가 로그인 사용자만 이용할 수 있는 페이지에 접근하려고 할 때, 서버는 `401 Unauthorized`를 반환합니다.

- _403 Forbidden_  
  _서버가 요청을 이해했지만, 요청을 처리할 권한이 없을 때 사용됩니다._
  `403 Forbidden`은 인증 여부와 관계없이 클라이언트가 요청한 리소스에 접근할 권한이 없을 때 발생합니다.
  서버는 요청을 거부하며, 리소스를 제공하지도 않습니다.
  예를 들어, 로그인한 사용자라도 권한이 없는 페이지(관리자 페이지)에 접근할 때 서버는 `403 Forbidden` 상태 코드를 반환합니다.

- _404 Not Found_  
  _서버가 요청한 리소스를 찾을 수 없을 때 사용됩니다._
  `404 Not Found`는 클라이언트가 요청한 URL이나 리소스가 서버에 존재하지 않을 때 반환됩니다.
  이는 URL이 잘못되었거나, 요청된 리소스가 삭제되었거나, 서버가 해당 리소스를 찾을 수 없는 경우에 발생합니다.
  예를 들어, 존재하지 않는 웹 페이지 URL을 요청받을 때, 서버는 `404 Not Found` 상태 코드를 반환합니다.

- _408 Request Timeout_  
  _클라이언트의 요청이 너무 오래 걸려서 서버가 이를 처리할 수 없을 때 사용됩니다._
  `408 Request Timeout`은 클라이언트가 서버에 요청을 보내고 응답을 기다리는 동안 너무 오래 시간이 경과했을 때 발생합니다.
  이는 클라이언트 측에서 요청을 너무 느리게 보냈거나, 서버가 클라이언트의 요청을 충분히 받을 수 없을 때 발생합니다.
  예를 들어, 클라이언트가 너무 큰 데이터를 전송하거나, 네트워크가 불안정해서 요청을 너무 오래 처리한 경우 서버는 `408 Request Timeout`을 반환할 수 있습니다.

## 500번 대: 서버 오류 (Server Error)

[# 목차](#-목차)

_500번 대 상태 코드는 서버 측에서 발생한 문제를 나타냅니다._
이 오류들은 클라이언트가 요청을 제대로 보냈지만, 서버가 요청을 처리하는 동안 내부적인 문제가 발생했음을 의미합니다.
즉, 요청 자체에 문제가 있는 것이 아니라 서버가 요청을 처리할 수 없는 상황이 발생했을 때 반환됩니다.

### 500번 대 주요 상태 코드

- _500 Internal Server Error_  
  _서버에서 요청을 처리하는 중에 예기치 않은 오류가 발생했을 때 사용됩니다._
  `500 Internal Server Error`는 서버 자체에서 문제가 발생했음을 나타냅니다.
  이는 서버가 클라이언트의 요청을 처리하려고 시도했지만, 서버 내부에서 오류가 발생하여 요청을 완료할 수 없을 때 발생합니다.
  서버의 버그, 설정 오류, 데이터베이스 연결 문제 등이 원인일 수 있습니다.
  예를 들어, 서버 코드에 오류가 있어 클라이언트의 요청을 처리할 수 없을 때, 서버는 `500 Internal Server Error`를 반환할 수 있습니다.

- _502 Bad Gateway_  
  _서버가 게이트웨이 또는 프록시로서 요청을 처리하려 했으나, 상위 서버로부터 잘못된 응답을 받았을 때 사용됩니다._
  `502 Bad Gateway`는 보통 서버가 다른 서버(백엔드 서버, 데이터베이스 서버 등)와 통신 중 문제가 발생했을 때 반환됩니다.
  예를 들어, 서버가 다른 서버에 요청을 보냈지만, 그 서버가 유효한 응답을 보내지 않았을 때 발생합니다.
  웹 서버가 백엔드 API 서버에 요청을 보냈지만, 그 서버가 오류를 반환한 경우 `502 Bad Gateway`가 발생합니다.

- _503 Service Unavailable_  
  _서버가 일시적으로 과부하 되었거나 유지보수 중이라 요청을 처리할 수 없을 때 사용됩니다._
  `503 Service Unavailable`은 서버가 요청을 처리할 수 없는 상태일 때 발생합니다.
  서버가 과부하 상태거나, 유지보수 작업이 진행 중일 때 사용되며 일반적으로 일시적인 문제이며, 일정 시간이 지나면 복구될 수 있습니다.
  예를 들어, 서버가 너무 많은 트래픽을 처리해야 할 때, 서버가 처리 능력을 초과하면 `503 Service Unavailable` 상태 코드를 반환할 수 있습니다.

- _504 Gateway Timeout_  
  _서버가 게이트웨이 또는 프록시 역할을 하면서, 상위 서버로부터 적시에 응답을 받지 못했을 때 사용됩니다._
  서버가 다른 서버로부터 응답을 받으려고 했으나, 지정된 시간 내에 응답을 받지 못했을 때 발생합니다.
  이는 서버 간의 네트워크 문제나 응답 지연 등이 원인일 수 있습니다.
  예를 들어, 웹 서버가 백엔드 API 서버에 요청을 보냈으나, 그 서버가 응답을 주지 않아서 `504 Gateway Timeout` 오류가 발생할 수 있습니다.

- _505 HTTP Version Not Supported_  
  _서버가 클라이언트에서 요청한 HTTP 버전을 지원하지 않을 때 사용됩니다._
  `505 HTTP version Not Supported`는 클라이언트에서 서버가 지원하지 않는 HTTP 버전으로 요청을 보낼 때 발생합니다.
  예를 들어, 클라이언트가 HTTP/1.1 대신 HTTP/2 또는 더 이전 버전의 HTTP를 사용하여 서버에 요청할 경우 발생할 수 있습니다.
  클라이언트가 서버에 HTTP/2 프로토콜을 사용하여 요청을 보냈지만, 서버가 HTTP/1.1만 지원하는 경우 `505 HTTP Version Not Supported` 상태 코드가 반환될 수 있습니다.

## 마무리

[# 목차](#-목차)

HTTP 상태 코드는 클라이언트와 서버 간의 요청 및 응답 흐름에서 중요한 역할을 합니다.
각 상태 코드가 의미하는 바를 잘 이해하고 있으면, 개발자나 시스템 관리자들은 문제 발생 시 더 빠르게 대응할 수 있습니다.
또한, 올바른 상태 코드를 반환하는 것은 웹 애플리케이션의 품질을 높이는 데 중요한 요소로, 사용자 경험을 향상하는 데에도 큰 도움이 됩니다.
HTTP 상태 코드를 이해하고 적절하게 활용하는 것은 웹 개발에서 반드시 알아야 할 기본적인 지식이므로, 이를 잘 숙지하여 효율적인 웹 시스템을 구축할 수 있습니다.
